# MVP 方案问题解决提示词集合

本文档包含针对巴西博彩平台 MVP 测试方案中每个待解决问题的专用提示词。每个提示词都包含项目背景、问题澄清和解决建议三个部分。

## 提示词 1: 文档架构设计

### 使用场景
当你需要设计或优化文档架构,将现有的六层结构映射到 DocuMind 的四层结构时使用。

### 提示词内容

```
# 项目背景

我正在为一个巴西博彩平台建立基于 DocuMind 理念的产品文档体系。DocuMind 是一个 AI 驱动的智能产品文档协作平台,其核心理念是将设计作为事实源,通过自动化流程将 Figma 设计稿转化为结构化的产品文档。

DocuMind 采用四层文档结构:
1. **模块层**: 功能模块的整体描述
2. **页面层**: 具体页面的设计和交互
3. **组件层**: 可复用的 UI 组件
4. **API 层**: 后端接口和数据源

我们的博彩平台技术栈较为复杂,原本设计了六层结构:
- 三方数据层(sportRader)
- 服务层
- API/WebSocket 层
- 组件层
- 页面层
- 模块层

此外,我们还需要记录用户用例(交互)文档。

# 当前待解决问题和矛盾

**问题**: 六层结构与 DocuMind 的四层结构不一致,可能导致概念混淆和学习成本增加。

**矛盾点**:
1. 我们的六层结构更细致地区分了后端技术栈(三方数据、服务、API/WebSocket),但 DocuMind 只有一个 API 层
2. 用户用例文档在 DocuMind 的四层结构中没有明确的位置
3. 如果强行合并层级,可能会丢失技术细节;如果保持六层,又偏离了 DocuMind 的理念

**影响**:
- 团队成员需要理解两套不同的概念体系
- 不利于使用 DocuMind 的最佳实践和工具
- 可能导致文档组织混乱,维护困难

# 建议

请帮我设计一个优化的文档架构方案,要求:

1. **保持与 DocuMind 四层结构的一致性**,便于团队理解和使用
2. **不丢失技术细节**,能够清晰表达博彩平台的复杂技术栈
3. **合理安排用户用例文档**,使其成为文档体系的有机组成部分
4. **提供具体的映射关系**,说明如何将原有六层结构转换为新的架构
5. **给出实际示例**,例如"体育投注模块"如何在新架构下组织文档

请提供:
- 优化后的四层文档架构设计
- 每一层的职责和包含的内容
- 原有六层结构到新架构的映射关系
- 用户用例文档的安排方式
- 至少一个完整的模块示例,展示四层文档的组织结构
```

## 提示词 2: 文档模板标准化设计

### 使用场景
当你需要为每个文档层级设计标准化的 Markdown 模板时使用。

### 提示词内容

```
# 项目背景

我正在为一个巴西博彩平台建立基于 DocuMind 理念的产品文档体系。文档采用四层结构:模块层、页面层、组件层、API 层。所有文档使用 Markdown 格式存储在 Git 仓库中。

博彩平台的特点:
- **业务复杂**: 包含体育投注、真人娱乐场、老虎机游戏、账户管理等多个模块
- **技术栈复杂**: 涉及前端组件、后端 API、WebSocket 实时通信、三方数据源(sportRader)
- **合规要求高**: 需要详细记录用户故事、交互流程、数据流向等
- **团队协作**: 设计师、产品经理、前端开发、后端开发需要共同维护文档

我们计划使用 AI(Manus)来辅助生成文档,因此文档模板需要:
- 结构清晰,便于 AI 理解和生成
- 字段标准化,便于自动化处理
- 包含必填和可选字段的明确定义
- 支持文档之间的关联引用

# 当前待解决问题和矛盾

**问题**: 缺少各层级文档的标准模板和字段定义,导致文档质量参差不齐。

**具体表现**:
1. 不同人员编写的文档格式不统一,有的详细有的简略
2. 缺少必要的元信息(如负责人、状态、最后更新时间等)
3. 文档之间的关联关系不清晰,难以追溯
4. AI 生成的文档缺少统一的输出格式,需要大量人工调整

**矛盾点**:
- 模板太简单,无法包含足够的信息
- 模板太复杂,填写成本高,团队不愿意使用
- 需要平衡标准化和灵活性

**影响**:
- 文档质量不稳定,影响团队协作效率
- AI 辅助生成的效果不理想
- 难以进行自动化处理(如索引生成、搜索等)

# 建议

请为四个文档层级分别设计标准化的 Markdown 模板:

1. **模块文档模板**
   - 包含模块的基本信息、概述、包含的页面、技术架构、变更历史等
   - 支持关联到子页面文档
   
2. **页面文档模板**
   - 包含页面的基本信息、用户故事、页面状态、交互流程、使用的组件、调用的 API 等
   - 支持嵌入流程图(Mermaid)
   - 支持关联到组件和 API 文档
   
3. **组件文档模板**
   - 包含组件的基本信息、Figma 链接、组件属性(Props)、变体、状态、使用示例等
   - 支持嵌入可交互的 HTML 组件预览
   - 支持关联到 API 和使用该组件的页面
   
4. **API 文档模板**
   - 包含 API 的基本信息、请求参数、响应格式、错误码、调用示例等
   - 支持区分不同类型的 API(REST API、WebSocket、三方数据源)
   - 支持关联到调用该 API 的组件和页面

每个模板请提供:
- 完整的 Markdown 模板代码
- 必填字段和可选字段的说明
- 字段的格式规范(如日期格式、状态枚举值等)
- 文档关联的语法(使用 `@` 符号)
- 至少一个填写示例
```

## 提示词 3: Figma 数据提取方案设计

### 使用场景
当你需要设计从 Figma 提取组件数据的技术方案时使用。

### 提示词内容

```
# 项目背景

我正在为一个巴西博彩平台建立基于 DocuMind 理念的产品文档体系。DocuMind 的核心理念是"设计即事实源",即将 Figma 设计稿作为产品文档的源头,通过自动化流程将设计信息同步到文档中。

我们的工作流程:
1. 设计师在 Figma 中设计组件(投注卡片、赔率按钮、游戏卡片等)
2. 使用自动化脚本从 Figma 提取组件信息
3. 使用 AI(Manus)根据 Figma 数据生成组件文档
4. 产品经理审核和完善文档
5. 使用 AI 生成可交互的 HTML 组件展示

技术环境:
- Figma 设计文件包含多个 Page,每个 Page 对应一个功能模块
- 组件使用 Figma 的 Component、Variant、Properties 功能
- 需要提取的信息包括:组件名称、描述、属性、变体、截图等
- 开发环境为 Node.js,可以使用 JavaScript/TypeScript

# 当前待解决问题和矛盾

**问题**: Figma 数据提取和同步机制不明确,无法实现自动化的"设计到文档"流程。

**具体困惑**:
1. 应该使用 Figma REST API、Figma Plugin API,还是其他方式?
2. 如何获取组件的详细信息(属性、变体、样式等)?
3. 如何获取组件的截图或预览图?
4. 如何处理组件的层级关系(父组件、子组件)?
5. 如何识别组件的变更(新增、更新、删除)?
6. 如何应对 Figma API 的速率限制?

**矛盾点**:
- Figma REST API 功能强大但有速率限制
- Figma Plugin API 可以获取更多细节但需要手动触发
- 完全自动化同步可能导致频繁调用 API 被限流
- 手动触发同步又失去了自动化的优势

**影响**:
- 无法实现"设计到文档"的自动化流程
- 可能需要大量手动工作来录入 Figma 信息
- 设计变更后文档无法及时同步

# 建议

请帮我设计一个完整的 Figma 数据提取方案,要求:

1. **明确技术选型**
   - 选择合适的 Figma API(REST API / Plugin API)
   - 说明选择的理由和优缺点
   
2. **设计数据提取流程**
   - 如何认证和连接 Figma
   - 如何遍历文件和组件
   - 如何提取组件的详细信息
   - 如何获取组件截图
   
3. **设计数据存储格式**
   - 提取的数据应该保存为什么格式(JSON/YAML/其他)?
   - 数据结构应该如何设计?
   
4. **设计同步机制**
   - 应该采用定期自动同步还是手动触发?
   - 如何识别组件的变更?
   - 如何处理 API 速率限制?
   
5. **提供实现示例**
   - 提供一个简单的 Node.js 脚本示例
   - 展示如何调用 Figma API 获取组件列表
   - 展示如何将数据保存为 JSON 文件

请提供详细的技术方案和代码示例。
```

## 提示词 4: AI 提示词模板设计(生成组件文档)

### 使用场景
当你需要设计用于生成组件文档的 AI 提示词模板时使用。

### 提示词内容

```
# 项目背景

我正在为一个巴西博彩平台建立基于 DocuMind 理念的产品文档体系。我们使用 AI(Manus)来辅助生成组件文档,工作流程如下:

1. 使用脚本从 Figma 提取组件信息,保存为 JSON 格式

组件文档的标准模板包含以下章节:
- 基本信息(组件 ID、Figma 链接、状态等)
- 组件概述(功能描述和使用场景)
- 组件预览(嵌入可交互 HTML)

  ],
  "variants": [
    {"name": "默认样式", "thumbnail": "url1"},

**具体表现**:
1. AI 生成的文档经常缺少某些章节
2. 基本信息填充不准确,例如日期格式不对
3. 组件属性表格格式混乱
4. 生成的内容过于简单或过于冗长
5. 没有生成文档关联(@ 引用)
6. 有时会生成不相关的内容

**矛盾点**:
- 提示词太简单,AI 生成的内容不够准确
- 提示词太复杂,AI 可能理解错误或遗漏信息
- 需要平衡灵活性和标准化

**影响**:
- 人工审核和修改的工作量大
- 文档质量不稳定
- 影响整体效率

# 建议

请帮我设计一个高质量的 AI 提示词模板,用于生成组件文档,要求:

1. **提供充分的上下文**
   - 说明任务目标和背景
   - 说明文档的用途和读者
   
2. **明确输入格式**
   - 说明 Figma 数据的 JSON 结构
   - 说明文档模板的 Markdown 格式
   
3. **明确输出要求**
   - 严格按照模板格式输出

4. **提供示例**

   - 如果 Figma 数据缺少某些字段,应该如何处理?

## 提示词 5: AI 提示词模板设计(生成可交互 HTML 组件)

### 使用场景
当你需要设计用于生成可交互 HTML 组件的 AI 提示词模板时使用。

### 提示词内容

```
# 项目背景

我正在为一个巴西博彩平台建立基于 DocuMind 理念的产品文档体系。我们需要为每个组件生成可交互的 HTML 文件,嵌入到组件文档中,方便团队查看组件的不同状态和变体。

工作流程:
1. 产品经理完善组件文档(基于 AI 生成的草稿)
2. 使用 AI(Manus)根据组件文档生成可交互的 HTML 文件
3. 将 HTML 文件保存到组件文档同级目录
4. 在组件文档中通过 iframe 嵌入 HTML 文件

HTML 组件的要求:
- 使用原生 HTML/CSS/JavaScript,不依赖框架
- 使用 TailwindCSS CDN 进行样式设计
- 响应式设计,适配不同屏幕尺寸
- 提供交互控制面板,允许用户切换属性、状态、变体
- 尽可能还原 Figma 设计稿的样式

组件文档示例:
```markdown
# 投注卡片

## 基本信息
- **组件 ID**: `component_bet_card`
- **Figma 链接**: [查看设计稿](https://figma.com/...)

## 组件概述
投注卡片用于展示单个投注选项,包括赛事信息、赔率和投注按钮。

## 组件属性 (Props)
| 属性名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|------|--------|------|
| eventName | string | 是 | - | 赛事名称 |
| odds | number | 是 | - | 赔率 |
| status | string | 否 | "正常" | 状态(正常/禁用) |

## 组件变体
### 变体1: 默认样式
正常可点击状态

### 变体2: 禁用状态
赔率变灰,不可点击

## 组件状态
- **正常**: 白色背景,黑色文字
- **悬停**: 浅蓝色背景
- **点击**: 蓝色背景,白色文字
- **禁用**: 灰色背景,灰色文字,不可点击
```

# 当前待解决问题和矛盾

**问题**: HTML 组件生成的技术方案和实现逻辑不明确,导致生成的组件质量不稳定。

**具体表现**:
1. 生成的 HTML 文件无法正常运行(缺少必要的标签或脚本)
2. 样式与 Figma 设计稿差异较大
3. 交互控制面板功能不完整或不可用
4. 没有实现响应式设计
5. 代码质量差,缺少注释,难以维护

**矛盾点**:
- 完全还原 Figma 设计需要大量细节,AI 可能无法准确实现
- 使用框架(React/Vue)可以提升开发效率,但增加了依赖和复杂度
- 提供过多的交互控制会让界面复杂,但控制太少又无法展示组件的所有状态

**影响**:
- 生成的 HTML 组件无法直接使用,需要大量人工修改
- 组件展示效果不理想,影响团队对设计的理解
- 降低了文档的实用性

# 建议

请帮我设计一个高质量的 AI 提示词模板,用于生成可交互的 HTML 组件,要求:

1. **明确技术要求**
   - 使用原生 HTML/CSS/JavaScript
   - 使用 TailwindCSS CDN
   - 响应式设计
   - 所有代码内联在一个 HTML 文件中
   
2. **明确交互要求**
   - 提供控制面板,包含所有可配置的属性
   - 支持切换组件状态(正常/悬停/点击/禁用)
   - 支持切换组件变体(如果有多个)
   - 属性修改后组件实时更新
   
3. **明确样式要求**
   - 尽可能还原 Figma 设计稿
   - 使用 TailwindCSS 的 utility classes
   - 提供清晰的布局(控制面板在顶部,组件展示在下方)
   
4. **明确代码质量要求**
   - 代码结构清晰,有良好的注释
   - 使用现代 JavaScript 语法(ES6+)
   - 变量和函数命名语义化
   
5. **提供完整示例**
   - 提供一个完整的输入示例(组件文档)
   - 提供一个完整的输出示例(HTML 文件代码)
   - 确保示例可以直接在浏览器中运行

请提供完整的提示词模板,可以直接复制使用。
```

## 提示词 6: 文档关联机制设计

### 使用场景
当你需要设计文档之间的关联机制和引用语法时使用。

### 提示词内容

```
# 项目背景

我正在为一个巴西博彩平台建立基于 DocuMind 理念的产品文档体系。文档采用四层结构(模块→页面→组件→API),存储在 Git 仓库中,使用 Markdown 格式。

DocuMind 的一个重要特性是文档之间的双向关联,通过 `@` 符号可以引用其他文档,形成知识图谱。例如:
- 页面文档可以引用使用的组件: `[@投注卡片](../components/bet-card.md)`
- 组件文档可以引用被使用的页面: `[@赛事列表页](../pages/event-list.md)`
- 组件文档可以引用调用的 API: `[@获取赛事列表](../apis/get-events.md)`

我们的文档仓库结构:
```
modules/
├── sports-betting/
│   ├── README.md (模块文档)
│   ├── pages/
│   │   ├── event-list.md
│   │   └── event-detail.md
│   ├── components/
│   │   ├── bet-card.md
│   │   └── odds-button.md
│   └── apis/
│       ├── get-events.md
│       └── place-bet.md
└── account-management/
    └── ...
```

# 当前待解决问题和矛盾

**问题**: 文档关联机制不够明确,难以追踪组件的设计来源和实现状态,不利于版本管理。

**具体困惑**:
1. 如何在 Markdown 中表示文档关联?使用相对路径还是绝对路径?
2. 如何确保关联的准确性?如果被引用的文档被删除或移动,如何处理?
3. 如何实现双向关联?例如组件文档引用了页面,页面文档是否需要反向引用组件?
4. 如何在文档网站中展示关联关系?点击关联链接应该跳转到哪里?
5. 如何通过唯一标识符(ID)关联文档、Figma 组件、HTML 组件?

1. 设计师在 Figma 中设计和修改组件

3. AI 自动更新组件文档
4. 产品经理审核变更
5. 提交到 Git 仓库,记录版本历史

设计变更的类型:
- **新增组件**: Figma 中新增了组件,需要生成新的文档
- **更新组件**: Figma 中组件的属性、变体、样式发生变化,需要更新文档
- **删除组件**: Figma 中删除了组件,需要标记或删除文档
- **重命名组件**: Figma 中组件重命名,需要更新文档和关联关系

版本管理需求:
- 能够追溯每个组件的变更历史
- 能够对比不同版本的差异
- 能够回滚到历史版本
- 能够关联 Figma 版本和文档版本

**问题**: 版本同步机制不明确,当 Figma 中的组件更新后,难以同步到文档和 HTML 文件,无法追踪变更历史。

3. 如何对比新旧版本的差异?
4. 如何在文档中记录变更历史?
5. 如何关联 Figma 版本号和文档版本号?
6. 如果组件被删除,文档应该删除还是标记为"已废弃"?
7. 如果组件重命名,如何更新所有引用该组件的文档?

**矛盾点**:
- 自动同步可以确保实时性,但可能导致频繁变更,影响稳定性
- 手动同步可以控制变更节奏,但可能遗漏变更或延迟同步
- 完全自动更新文档可能覆盖人工修改的内容
- 需要人工审核又增加了工作量

**影响**:
- 设计变更后文档无法及时同步,导致文档与设计不一致
- 无法追溯组件的变更历史
- 难以进行版本管理和回滚
- 影响团队协作效率

   - 如何关联 Figma 版本号和文档版本号?

5. **冲突处理机制**
   - 如果 Figma 更新了组件,但文档也被人工修改了,如何处理冲突?
   - 是否需要提供 Diff 对比和合并功能?

   - 如何生成变更报告?
